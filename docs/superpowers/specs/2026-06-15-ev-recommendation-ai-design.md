# 全レース予想 + EV提案AI 設計

## 目的
任意のJRAレースに対し、**統計ベースの予想**と**期待値(EV)に基づく買い目提案**を、
それぞれ**確信度つき**で必ず提示するAI。利益を保証するものではなく、
「較正された予測」と「今あるオッズで最もEVの高い買い方」を正直に提示する。

## 背景(なぜこの形か)
診断(`diagnose_edge.py` / `diagnose_noodds.py`)で次が判明している:

- 学習済みモデルは過去成績などの統計から**本物の予測力**を持つ
  (Brier 0.0565、全分位で予測勝率≒実勝率の**較正済み**)。
- ただしその予測は**単勝オッズより正確ではない**(市場 Brier 0.0566 と同等、
  オッズ抜きモデルは 0.0660 と劣る)。単勝フラット張りでの利益エッジは無い。

結論:**予測そのものには価値がある**(確信度つきで見せるべき)。一方で利益は
約束できない。単勝市場と予測が食い違う部分は、**連系・複系プール
(複勝/ワイド/馬連)が単勝市場と食い違ったとき**に+EVとして現れうるので、
予測の提示とEV提案は地続きである。

## 出力(レースごとに必ず2部構成)

### ① 統計ベース予想(全馬)
モデルが過去成績・馬体・馬場・距離などから算出した**推定勝率・複勝率の
ランキング**。各馬に**確信度**を併記。

### ② EV提案(単勝 + 複勝・ワイド・馬連)
①の確率 × 実オッズ = EV。EV降順に最良の買い目を**EVと確信度**つきで提示。
+EV(EV ≥ 0、すなわち期待回収率 ≥ 100%)が一つも無ければ「見送り推奨」と
明記しつつ、最良候補は必ず表示する。

## 確信度(confidence)の定義
"市場に勝てる自信" ではなく、**その推定値そのものの信頼度**。低/中/高 + 0–1 数値。
構成要素(説明可能・加重平均):

- **データ量**: 対象馬の過去走数(`prev_runs`)。初出走・少データほど低い。
  連系は関与する全馬の最小データ量を採用。
- **確率の鋭さ**: 抜けた本命がいるか(上位確率の集中度)。混戦ほど低い。
- **頭数・オッズ確定**: 出走頭数の妥当性、`win_odds` が確定済み(None でない)か。

確信度は予測・EVの**大きさとは独立**に出す(「どれくらい良いか」と
「その数字をどれだけ信用していいか」を分離して提示)。

## コンポーネント(疎結合・各々テスト可能)

### 1. 確率レイヤー `keiba/predictor.py`(拡張)
較正済み勝率(既存 `win_probabilities(model)`)を起点に、ハーヴィル法で:
- **複勝確率** `place_probabilities(win, k)`: 既存(k=2)を JRA ルールに合わせて
  使い分ける。`place_k(field_size)`: 8頭以上=3、5–7頭=2、≤4頭=複勝なし(None)。
- **馬連確率** `quinella_probabilities(win)`: 2頭がともに上位2着(順不同)。
- **ワイド確率** `wide_probabilities(win)`: 2頭がともに上位3着(順不同)。

ハーヴィルは順列ベース。頭数が多いと全順列は重いので、
**上位N頭(例: 確率上位12頭)に絞って**ペア確率を近似計算する。

### 2. オッズ取得 `keiba/odds_page.py`(新規・最大の技術リスク)
netkeiba のオッズ JSON API から複勝(範囲オッズ)・ワイド・馬連を取得・パース。
- 取得関数(ネットワーク。テスト対象外)と、パース関数(**JSONフィクスチャで
  TDD**)を分離。`scraper.py` / `result_page.py` の `fetch_html` 分離方針に倣う。
- 返り値:
  - 複勝: `{馬番: (下限, 上限)}`
  - ワイド/馬連: `{(馬番a, 馬番b): オッズ}`(キーは昇順タプルで正規化)
- 馬番↔馬名の対応は `Race.horses` から引く。

### 3. EVエンジン `keiba/expected_value.py`(拡張)
`ev(prob, odds) = prob * odds - 1.0`(既存 win_ev/place_ev を一般化)。
- 複勝の範囲オッズは**下限採用 = 保守的EV**。

### 4. 確信度 `keiba/confidence.py`(新規)
`confidence(...) -> (score: float, level: str, reason: str)`。
上記の要素から 0–1 のスコアと 低/中/高、短い理由文を返す純粋関数。

### 5. 推奨集約 `keiba/recommend.py`(拡張)
単勝/複勝/ワイド/馬連の全候補について EV と確信度を計算し、**券種横断で
EV降順**(同点は確信度降順)にランキング。`recommend_all(race, win_probs,
odds) -> list[Bet]`。各 `Bet` は券種・対象・確率・オッズ・EV・確信度・理由を持つ。
+EV の有無を判定し、無ければ「見送り推奨」フラグを立てる。

### 6. CLI `main.py`(結線)
レースごとに ①予想ランキング ②EV提案 を確信度つきで表示。

## データフロー
```
scrape_race → build_features → win_probabilities(model)
  → predictor: 複勝/馬連/ワイド確率(ハーヴィル)
  → odds_page: 複勝/ワイド/馬連オッズ取得
  → expected_value: 各候補のEV
  → confidence: 各候補の確信度
  → recommend: 券種横断ランキング + 見送り判定
  → main: ①予想 ②EV提案 を表示
```

## テスト(TDD)
- **純粋関数を先に**: ハーヴィル(複勝/馬連/ワイド確率)、EV、確信度、ランキング・
  見送り判定。ネットワーク不要で単体テスト。
  - 確率の健全性: 各確率が [0,1]、複勝率 ≥ 勝率、ワイド率 ≥ 馬連率 など。
- **odds_page**: 保存した JSON フィクスチャでパースを検証。
- 既存49テストを壊さない。

## 進め方(リスク順)
1. 確率レイヤー(複勝k使い分け・馬連・ワイド)+ テスト
2. EVエンジン拡張 + 確信度 + テスト
3. 推奨集約(券種横断ランキング・見送り判定)+ テスト
4. odds_page(フィクスチャTDD → ネットワーク取得)
5. main.py 結線・実レースで動作確認

純粋コア(1–3)を先に完成させ、ネットワーク依存(4)を最後に統合する。

- `python main.py <race_id>` で、確定オッズのある直前レースに対し
  ①予想 ②EV提案 が確信度つきで表示されることを確認
