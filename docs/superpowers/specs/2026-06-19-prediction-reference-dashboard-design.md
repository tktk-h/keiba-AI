# 予想の資料ダッシュボード(スマホWeb) — 設計

日付: 2026-06-19

## 背景と目標
一連の検証で「自動で市場に勝つ(回収率>100%)」は現データでは不可能と確定した
(市場はほぼ完璧にキャリブレート、モデル/特徴量/買い方を変えても控除率の壁~80%)。
そこで目標を**転換**する:

> ツールは利益機械ではなく、**人間(ユーザー)の予想を補助する「資料」**になる。
> **最終目標は、競馬場でスマホから使えること。**

ユーザーが選んだ価値: ①市場との乖離ハイライト ②各馬の一覧ダッシュボード
③予想根拠の提示(乖離馬のみ)。**主役の提供形態はスマホ向けWeb**(既存Flaskアプリの
レスポンシブ化)。CLI(`main.py`)は開発時の確認用として同じ表示内容を出す。

## 提供形態とフェーズ
予想ロジックは `keiba/report.py` ほか共有レイヤに集約済みで、CLIとWebは薄い表示層。
そこで段階的に進める:
- **フェーズ1(本スペックの中核): 共有ロジック** — `deviation_score`・`explain.py`・
  `build_race_report` への付与。表示面に依存しない「頭脳」。CLI表示も同時に更新(低コスト)。
- **フェーズ2: スマホWeb表示** — `web/templates/race.html` をモバイル優先のレスポンシブに。
  フェーズ1の出力をそのまま描画。
- **フェーズ3(別途): 公開方法** — LAN→トンネル(ngrok等)→クラウド常駐のいずれか。
  アプリのコードに依存しないため後決めでよい(競馬場利用にはクラウド/トンネルが必要)。

## スコープ
- 新規 `keiba/explain.py`: ロジスティック回帰の線形寄与アトリビューション(純粋・TDD)。
- `keiba/recommend.py` に純粋関数 `deviation_score` を**追加**(既存ロジックは変更しない)。
- `build_race_report` で各馬に評価スコアを、乖離馬に `reasons` を付与(CLI/Web共通)。
- `main.py`(CLI)と `web/templates/race.html`(スマホWeb)の表示を新ダッシュボードに更新。
- モデル・データ収集・既存の予想/買い目ロジックは変更しない。

## 非目標 (YAGNI)
- モデルや特徴量の変更、新データ取得。
- 買い目で優位性を出すこと(出ないと確定済み)。
- フェーズ3の公開/デプロイ作業(本スペックでは設計のみ、実装は別)。
- ネイティブアプリ化(スマホブラウザで完結させる)。
- LGBM 等の非線形モデルの寄与(SHAP)。現行の本番モデルはロジスティック回帰なので
  線形寄与で十分。将来モデルを変えたら別途。

## 性能/UX(スマホ・競馬場での実用性)
- `--enrich`(各馬の過去走取得)は1レース数分かかる → スマホでは待ち時間が課題。
  - Web側は**ローディング表示**を出し、取得完了後に結果を描画(無言で固まらせない)。
  - `build_race_report` の結果を**短期キャッシュ**(同一race_idの再取得を避ける)。
  - 取得は重いので、フェーズ2で「先に出馬表だけ即表示→過去走は順次反映」等は
    将来の最適化候補(本スペックでは必須としない)。
- スマホ画面が狭いので、表は主要列に絞り、根拠は1〜2行に圧縮(下記デザイン参照)。

## 表示デザイン(CLI / スマホWeb 共通の内容)

### ① 一覧ダッシュボード(全馬)
予想勝率の高い順に1行ずつ:
`評価 / 馬名 / 予想勝率 / 複勝率 / 市場勝率 / 確信度`
- **予想勝率** = 本ツール(モデル)の推定勝率(=「あなたの考える勝率」)。
  市場勝率(オッズが示す勝率)と並べて比較できるようにする。
- **評価** = 市場との乖離を **-5〜+5 の11段階**で数値化(下記「乖離スコア」)。
  `+`ほど市場が過小評価(=妙味)、`-`ほど過大評価(=過剰人気)、`0`は互角。
- 評価の絶対値が大きい馬を視覚強調。

### 乖離スコア (-5〜+5, 11段階)
各馬のモデル勝率 `p` と市場勝率 `q` の対数比で算出:
```
score = clamp( round( ln(p / q) / ln(1.3) ), -5, +5 )
```
- `+n`: モデルが市場より高評価(市場に過小評価された馬)。+1≈1.3倍, +5≈3.7倍以上。
- `-n`: モデルが市場より低評価(市場に過大評価された馬)。
- `0`: 概ね互角(`p/q` が約 0.88〜1.14)。
- `p` または `q` が 0/欠損なら `0`(評価不能)。
- 基準の 1.3 は既存 `_value_tag` の `VALUE_FACTOR` を踏襲。
- 新規純粋関数 `deviation_score(model_p, market_p) -> int` を `keiba/recommend.py`
  に追加(TDD)。既存 `_value_tag` は表示から外すが関数は残置(後方互換)。

### ② 乖離馬の根拠(新規)
①で **|評価| ≥ 1**(市場と乖離)の馬についてのみ、評価を動かした
**非オッズ要因 上位2〜3個**を表示。
```
馬A (評価+4 予想勝率18.2%): 近走着順 ↑ ・ 馬体重 ↓
馬B (評価-3 予想勝率 6.1%): 出走間隔 ↓ ・ 年齢 ↓
```
- 各馬に **評価スコアと予想勝率**を併記。
- `↑` = その要因が勝率を押し上げた / `↓` = 押し下げた。
- 値はオッズに無い要因に限る(オッズ系6特徴量を除外)。

### ③ 買い目セクションの格下げ
現状の「② 買い目のEV(推奨)」を、優位性なしと明示する**参考扱い**に変更:
> 参考: +EV候補(市場効率につき控除率を越える優位性は確認されていません)

誤って「推奨」と読まれないよう文言で明確化。買い目の計算自体は現状維持。

## 新規モジュール `keiba/explain.py` (純粋関数)

```
MARKET_COLUMNS = {"win_odds","log_odds","odds_rank_pct","popularity",
                  "prev_avg_log_odds","prev_avg_popularity"}

feature_contributions(model, feature_row) -> dict[str, float]
    # model: Pipeline(StandardScaler, LogisticRegression)。
    # 各特徴量の対数オッズへの寄与 = coef_i * (x_i - mean_i)/scale_i。
    # feature_row: model.feature_columns_ をキーに持つ dict/Series。
    # 欠損/数値化不能はその特徴量を 0 寄与として扱う。

deviation_reasons(contributions, exclude=MARKET_COLUMNS, top_n=3)
    -> list[tuple[str, float]]
    # exclude を除外し、|寄与| の大きい順に最大 top_n、寄与が 0 の要因は出さない。

FEATURE_LABELS_JA: dict[str, str]   # 特徴量 -> 日本語ラベル(表示用)
```

設計理由: 表示している勝率(モデル)と根拠が**必ず同じモデルから**出るので、
資料として矛盾しない。線形モデルなので寄与は厳密に分解できる。

## 配線
- `build_race_report` は既に `df`(=build_features) と `model` を持つ。各 prediction 行へ
  `score`(乖離スコア -5..+5)を付与し、`|score|≥1` の行には `df` の該当馬の特徴量行から
  `feature_contributions` → `deviation_reasons` を計算して
  `reasons: list[(label_ja, direction)]` を付与する。
- これにより **CLI と Web は同じ `report` dict を描画するだけ**でよい(表示面に依存しない)。
- `assemble_report` は純粋関数のまま(付与は `build_race_report` 側、または
  model/features を受け取れるよう引数追加。実装時に最小変更で選択)。
- フェーズ1: `main.py` が新フォーマットで ①②③ を表示。
- フェーズ2: `web/templates/race.html` が同じ dict をモバイル優先レイアウトで描画
  (評価バッジ・予想勝率/市場勝率の対比・乖離馬の根拠)。`web/app.py` はキャッシュを挟む。

## エラー処理
- `model` が無い(model.pkl 不在)場合: 根拠セクションは出さず、①は市場ベース勝率で表示。
- 乖離馬が 0 頭: 「乖離している馬はありません(市場と概ね一致)」と表示。
- 特徴量行が見つからない/欠損: その馬の根拠はスキップ(クラッシュしない)。

## テスト (TDD)
- `keiba/explain.py`:
  - `feature_contributions` が coef×標準化値を返す(既知の係数/スケーラで検算)。
  - 欠損特徴量は 0 寄与。
  - `deviation_reasons` が除外集合を外し、|寄与|順 top_n、0寄与を除く。
  - ラベル表に主要特徴量が存在。
- `deviation_score` (`keiba/recommend.py`):
  - 互角(p≈q)で 0、p=1.3q で +1、p=q/1.3 で -1。
  - 極端な乖離で ±5 にクランプ。
  - p または q が 0 で 0(評価不能)。
- `build_race_report` が各 prediction に `score` を、`|score|≥1` に `reasons` を付与する
  (`assemble_report` のテストで dict 構造を担保)。
- 既存の `report`/`recommend` のテストを壊さない(全スイート緑)。
- CLI/Web の表示は薄い結合層なので手動確認 + 上記レポートテストで担保。

## 成功基準
- **フェーズ1**: `python main.py <race_id> --enrich` で、全馬一覧(評価±5・予想勝率・
  市場勝率・複勝率・確信度) + 乖離馬(|評価|≥1)の非オッズ根拠 + 参考扱いの買い目が表示。
  評価スコア・予想勝率・根拠がすべて同一モデル由来で矛盾しない。全テスト緑。
- **フェーズ2**: スマホのブラウザで同じ内容が、狭い画面でも読めるレイアウトで表示される。
- 公開方法(フェーズ3)は本スペックの完了条件に含めない。
